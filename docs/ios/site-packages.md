# Site-Packages & Frameworks on iOS

This page explains how ksproject handles Python packages and xcframeworks during iOS/macOS builds — specifically how the `.frameworks/` directory in site-packages is discovered, collected, and integrated into the Xcode project.

---

## How Site-Packages Are Installed

When you build for iOS, ksproject installs your project's Python dependencies **cross-compiled for the target platform** using `uv pip install` with platform-specific flags.

Each platform gets its own directory:

```
project_dist/xcode/site_packages/
├── iphoneos/              # Physical device (arm64)
│   ├── kivy/
│   ├── myapp/
│   └── ...
├── iphonesimulator/       # Simulator (arm64 or x86_64)
│   ├── kivy/
│   ├── myapp/
│   └── ...
└── macos/                 # macOS (universal)
    ├── kivy/
    ├── myapp/
    └── ...
```

These directories are bundled into the final `.app` as resources and loaded by the Python runtime at startup.

---

## The `.frameworks/` Convention

Some Python packages ship with **xcframeworks** — pre-compiled iOS/macOS dynamic libraries in Apple's universal framework format. These are distributed inside pip wheels under a `.frameworks/` directory.

### How It Works

1. **Package authors** include `.frameworks/*.xcframework` in their wheel (via custom build backends)
2. **pip installs** the wheel into site-packages, placing `.frameworks/` alongside the Python code
3. **ksproject detects** `.frameworks/` in site-packages after installation
4. **Frameworks are moved** from site-packages to `project_dist/xcode/Support/`
5. **Xcode project** references them via framework search paths

### Example Flow

A package `kivy-sdl2` ships wheels containing:

```
kivy_sdl2-0.1.0-cp313-cp313-ios_arm64.whl
└── .frameworks/
    ├── SDL2.xcframework/
    ├── SDL2_image.xcframework/
    ├── SDL2_ttf.xcframework/
    └── SDL2_mixer.xcframework/
```

After `uv pip install`:

```
site_packages/iphoneos/
├── .frameworks/
│   ├── SDL2.xcframework/
│   ├── SDL2_image.xcframework/
│   └── ...
└── kivy_sdl2/
    └── __init__.py
```

ksproject then moves them:

```
project_dist/xcode/Support/
├── Python.xcframework/        # From BeeWare
├── SDL2.xcframework/          # From kivy_sdl2 pip package
├── SDL2_image.xcframework/    # From kivy_sdl2 pip package
├── SDL2_ttf.xcframework/      # From kivy_sdl2 pip package
└── SDL2_mixer.xcframework/    # From kivy_sdl2 pip package
```

---

## Why `.frameworks/` Are Moved Out

xcframeworks cannot stay inside site-packages because:

1. **Xcode needs them at link time** — They must be in a known framework search path
2. **Code signing** — Frameworks must be in the app bundle's `Frameworks/` directory
3. **Size** — Keeping them in assets would bloat the app with duplicate binaries

After moving, the `.frameworks/` directory is **removed from site-packages** to prevent it from being bundled into the app's Python assets.

---

## Framework Merging Across Platforms

When building for both iOS device and simulator, ksproject collects frameworks from **all** platform slices and merges them:

```
site_packages/iphoneos/.frameworks/
├── MyLib.xcframework/
│   └── ios-arm64/
│       └── MyLib.framework/

site_packages/iphonesimulator/.frameworks/
├── MyLib.xcframework/
│   └── ios-arm64_x86_64-simulator/
│       └── MyLib.framework/
```

Both get copied into `Support/` — xcframeworks already handle multi-platform via their internal slice structure.

---

## How Frameworks Are Referenced in Xcode

The generated `project.yml` (XcodeGen spec) includes:

```yaml
settings:
  FRAMEWORK_SEARCH_PATHS:
    - "$(PROJECT_DIR)/Support"
  LD_RUNPATH_SEARCH_PATHS:
    - "@executable_path/Frameworks"
```

And a **copy frameworks** build phase that embeds all xcframeworks into the app bundle at `<App>.app/Frameworks/`.

## Pre-installed Frameworks

ksproject automatically provides these frameworks (not from pip):

| Framework | Source | Cached At |
|-----------|--------|-----------|
| Python.xcframework | BeeWare CPython for Apple | `~/.kivyschool/apple_support/` |

The Python xcframework contains both iOS and macOS slices and is shared across all projects.

### SDL2 Frameworks from Kivy

The Kivy SDL2 xcframeworks are fetched from the KivySchool Anaconda channel as a pip package (`kivy_sdl2`):

```
~/.kivyschool/apple_support/kivy_sdl2/
├── SDL2.xcframework/
├── SDL2_image.xcframework/
├── SDL2_ttf.xcframework/
└── SDL2_mixer.xcframework/
```

## Creating Packages with Frameworks

If you're building a pip package that provides iOS frameworks, structure your wheel like this:

```
my_native_lib-1.0.0-cp313-cp313-ios_arm64.whl
├── my_native_lib/
│   ├── __init__.py
│   └── wrapper.py
└── .frameworks/
    └── MyNativeLib.xcframework/
        ├── ios-arm64/
        │   └── MyNativeLib.framework/
        │       ├── MyNativeLib        # Binary
        │       ├── Headers/
        │       └── Info.plist
        └── ios-arm64_x86_64-simulator/
            └── MyNativeLib.framework/
                └── ...
```

When users install your package in a ksproject app, the framework will be automatically:

1. Installed into site-packages with `.frameworks/`
2. Detected and moved to the Xcode project's `Support/` directory
3. Linked and embedded in the final `.app` bundle

## Comparison: Android vs iOS Native Assets

| Aspect | Android | iOS |
|--------|---------|-----|
| Native library format | `.so` (shared objects) | `.xcframework` (universal bundles) |
| Location in wheel | `.libs/<abi>/*.so` | `.frameworks/*.xcframework/` |
| Destination in project | `jniLibs/<abi>/` | `Support/` → `Frameworks/` |
| Discovery mechanism | Scan site-packages after install | Scan site-packages after install |
| Multi-arch handling | Separate dirs per ABI | xcframework contains all slices |

---

## Summary

The `.frameworks/` convention enables pip packages to deliver native iOS/macOS code alongside Python packages. ksproject's build system:

1. **Installs** packages with `uv pip install` (cross-compiled for target platform)
2. **Scans** for `.frameworks/` in the installed site-packages
3. **Moves** xcframeworks to the Xcode project's `Support/` directory
4. **Configures** framework search paths and embedding in the Xcode project
5. **Removes** `.frameworks/` from site-packages to avoid duplication in assets

This creates a seamless experience where `pip install some-ios-library` automatically makes native frameworks available in your mobile app — no manual Xcode configuration required.
